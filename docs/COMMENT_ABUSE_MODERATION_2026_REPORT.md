## 摘要（Abstract）

本報告系統性記述 Matters 平台於 2026 年針對「留言濫用」（comment abuse，涵蓋色情招攬、商業垃圾、博弈推廣、詐騙與機械式灌水）所建構之偵測與處置體系。該體系之發展具有三條交織的主線：(1) **群眾參與的治理機制**——以「守望相助」（Community Watch）為核心，將具備權限的社群成員納入內容處置流程，並以站方覆核為制衡；(2) **自動化偵測與分級處置**——以專用留言垃圾模型對每則留言評分，結合「海巡」自動代理（coastguard bot）之 ring（重複貼文）行為偵測，並於伺服器端建立「複合判定—分層告警」框架；(3) **訓練資料留存與模型矯正**——建立被動／主動／組裝三層訓練資料管線，並完成一次關鍵的標籤偏差矯正與模型重訓。

本報告之核心論點為：留言濫用處置不是單純的分類準確度問題，而是一個**價值對齊與正當程序**問題。在一個以華語深度與異議寫作為核心價值的平台，處置系統的「誤殺方向」與「用戶救濟途徑」，比其原始準確率更具決定性。本報告逐節記述每一項重大設置、資料截取、群眾參與機制與模型更新，並彙整方法論上踩過的雷，最後從平台營運與用戶權益雙重立場提出治理原則。

## 1. 引言（Introduction）

### 1.1 問題脈絡
任何開放的內容平台都面臨垃圾與濫用內容的壓力：色情招攬、外送茶、博弈推廣、賣帳號、詐騙與 SEO 內容農場。Matters 的特殊性在於，其存在理由之一是**保護在他處會被消音的華語表達**——政治評論、調查報導、異議書寫、學術與創作。這構成一組內生的張力：反垃圾與反審查在技術上共用同一套機制（一個分數加一個門檻），在價值上卻彼此對立。一個會降低能見度或隱藏內容的自動系統，一旦判斷出錯，對一般平台只是少推幾則貼文，對 Matters 則可能是平台親手執行了它本欲對抗之事。

留言層的張力又較文章層更為尖銳：留言短、語境依賴強、互動即時，誤殺的可感知度高，且留言往往是社群互動的即時現場。因此，留言濫用處置體系的設計，自始即須同時回答兩個問題：**如何有效攔截真正的濫用**，以及**如何在攔截的同時不傷害正當表達、並為誤判提供救濟**。

### 1.2 範圍與方法
本報告聚焦「留言」層之濫用處置，並納入與之共用的基礎設施（模型服務、訓練資料管線）。所有事實陳述以 git 版本紀錄為據；凡涉及線上行為者，另以部署與設定紀錄佐證。為求可重現性，關鍵事件均附 commit 雜湊與日期。

## 2. 系統前史與基礎（2024–2025）

留言層的處置體系並非憑空而生，而是建立在既有的文章層反垃圾基礎之上。

- **2024-08**：平台建立文章層反垃圾基礎設施（`matters-server`）。包含：於資料庫 schema 加入 spam 相關欄位與 API 轉接層（`2b09baf18`）、在發佈／修訂時觸發 spam 偵測（`a1438252e`）、於 `feature_flag` 表加入帶門檻的 spam 偵測開關（`12c5e831b`），並據分數將文章排除出熱門、最新、標籤、搜尋與活動文章流（`3cc2ea5cc`、`5eb4d4d2a`、`84dc30db4`、`3d9263773`、`893336f1c`）。同月並建立 OSS 端的 spam 狀態查詢、`setSpamStatus` 人工標註介面與 `isSpam` 篩選（`4ee768ff1`、`56a84ae84`）。
- **模型服務（`spam-detection-serverless`）**：2024-08 建立 SAM 專案骨架（`b4b1619`）；2025-01 重構並更新 spam 模型（`dc4c209`，此 commit 同時為日後 HF token 洩漏之源，見 §9）；2025-08 至 2025-12 持續迭代訓練 notebook 與推論程式（`434938d`、`f143163`、`41e87a0`）。

此一階段確立了兩項日後留言層沿用的關鍵設計：**「以分數加可調門檻決定能見度」的處置範式**，以及**「不刪除、只是不再被看見」的克制原則**。留言層初期即以此文章／短內容模型作為退化備援（fallback），待專用留言模型就緒後再行切換。

## 3. 群眾參與的治理機制：守望相助（Community Watch）

2026 年留言濫用處置最具制度意義的進展，是將**群眾參與**正式納入處置流程。其載體為「守望相助」（Community Watch），於 2026 年 5 月分階段建置並上線。

### 3.1 設計與分階段建置
- **2026-05-09**：加入守望相助 feature flag（`cbd2e015e`），以權限旗標界定可參與處置的社群成員。
- **2026-05-10（Phase 1–2，PR #4762/#4763）**：建立守望相助留言移除 API（`7136cf590`）、於留言上揭露守望相助處置動作（`032d829ab`）、建立公開審計查詢（`1e79a0302`、`4a0117eda`），並以 GraphQL 測試覆蓋（`5db47f566`）。
- **2026-05-11（Phase 3–5，PR #4764/#4765/#4769/#4771）**：將查詢重構至 connector 層（`b3543d34f`、`1accf9e71`），並新增**站方覆核動作**（staff review actions，`39c296535`）——此為群眾處置之制衡層。
- **2026-05-13 至 05-16**：修正審計讀取一致性（自主庫讀取，`12dd268b6`）、於留言列表保留守望相助佔位（`4c3efe67c`）、徽章與 feature flag 同步及回填（`60e151251`、`60d0e5ecb`）、通知覆核結果（`167377ece`）、對齊內容保留政策（`16f0ebeaa`）。
- **2026-05-17**：**守望相助正式發佈至 production**（`4e0d9a91f`，PR #4772）。
- **2026-05-18**：修正還原狀態（`0151e1b6b`）並揭露來源留言 URL（`738f5490f`）。

### 3.2 機制與制衡
守望相助的運作可歸納為「**群眾發動、站方制衡、全程可審計、可逆且可救濟**」：

1. **群眾發動**：具守望相助旗標之成員可對留言發動移除（`communityWatchRemoveComment`）。經查證，該動作將留言狀態設為 `banned`（隱藏而非刪除），並**自動同步寫入一筆檢舉**（reason 為 `illegal_advertising`，見 `communityWatchRemoveComment.ts`）。
2. **站方制衡**：站方可對群眾處置進行覆核（維持／推翻／調整理由）。被推翻（reversed）之處置可還原。
3. **可審計**：所有處置與覆核均有公開審計查詢與徽章揭露，使群眾處置透明可究。
4. **與既有檢舉機制並存**：平台原有的站內檢舉（`submitReport`，機制可追溯至更早年份）構成另一條群眾訊號來源；守望相助是其上的進階授權層。

### 3.3 群眾訊號之資料閉環
- **2026-05-28**：建立守望相助檢舉同步（`7bdd38fb1`，PR #4824），並於 OSS 批次封存守望相助檢舉之相關帳號（`8a0f84959`，PR #4822）。此舉使群眾處置紀錄成為後續模型訓練與 ring 偵測的**參考語料**（見 §6.3、§7）。

守望相助的意義在於：它把處置權力的一部分下放給社群，同時以站方覆核與全程審計作為防止濫權與誤判的制衡，並使每一次群眾處置都沉澱為可供模型學習的標註資料——構成「群眾參與—站方治理—資料回流」的閉環。

## 4. 自動化留言垃圾偵測

群眾參與之外，2026 年 6 月起平台建立了自動化的留言垃圾偵測，以彌補人力覆蓋之不足。

### 4.1 專用留言模型
- **2026-06-03（`spam-detection-scaffold`）**：建立留言層的標註採集與基線評分腳本（`49d656b`）、e5-small + 邏輯迴歸的留言垃圾模型與服務（`b18e84c`），並建立獨立 Lambda stack 之部署 workflow（`a6f7d11`）與門檻校準（group-K-fold OOF，`bc84eef`）。部署採 GitHub OIDC、不存放 AWS 金鑰（`42235e4`）。
- **2026-06-04**：服務端同時接受 matters-server 的 JSON `{text}` 與 raw 兩種 payload（`e994a27`）。
- **2026-06-04（`matters-server`）**：將留言垃圾評分路由至**專用留言模型**（`a6772c481`），與文章／短內容模型分離；每則留言之分數寫入 `comment.spam_score`。此前留言暫以短內容模型評分作為退化備援。

留言模型部署為獨立 serverless endpoint（`4xa9d5l1gd`），與文章模型（`ps349rjby5`）解耦，使兩層可獨立迭代。

### 4.2 自動代理：海巡（Coastguard Bot）
- **2026-06-03（`matters-coastguard-bot`）**：建立「偵測→分級→檢舉／移除」之自動代理骨架（`fcbf1b8`），並以資料校準之雙門檻啟動（移除 0.80／檢舉 0.55，`07bd7a9`）。
- **2026-06-04 至 06-06**：分階段上線設計（以 repo variable 控制、附 runbook，`a8b9c54`）；修正 staging GraphQL URL（`6b1caa6`）；以登入優先＋cloudscraper 對抗 production 之 Cloudflare 挑戰（`79a4e8b`）；依 production 觀察將 Tier-1 檢舉門檻上調至 0.80（`9c0dd4a`）。
- **設計要旨**：海巡以保守為預設（staging 環境、DRY_RUN 預設開、移除預設關），並設多重煞車：reversed-rate 自動煞車（群眾／站方推翻率過高即停手）、每輪動作上限、以及可逆的處置（`restoreCommunityWatchComment` 加申訴）。

海巡之定位並非取代群眾，而是**以群眾處置歷史為師**（見 §6.3 ring gate），對人力難以即時覆蓋的重複性濫用進行補抓。

## 5. 處置機制之演進

留言濫用之處置手段於 2026 年 6 月快速演進，形成多層次的處置光譜。

### 5.1 站方告警：檢舉 → Telegram
- **2026-06-02 至 06-04**：建立新檢舉發生時向管理員 Telegram 群組告警之機制（`6870b23f6`），並將 Telegram 投遞重構為獨立佇列 worker（`5b18fe430`，PR #4830），以使告警 worker 與 API 主程式得以不同節奏部署。此一 `report-alert` SQS → worker → Telegram 之管線，日後成為留言三層分流告警之共用基礎設施（見 §5.4）。

### 5.2 原生自動折疊（auto-collapse）
- **2026-06-13**：於伺服器端加入受環境旗標控制之留言自動折疊（`69e2153ea`，PR #4843）。當留言分數達系統門檻且作者非白名單時，狀態由 `active` 轉為 `collapsed`（折疊不刪，對應「不刪除只是不再被看見」）。預設關閉；經本期評估其純分數判定精準度不足（見 §6.2），**決定維持關閉**。

### 5.3 處置理由與資料庫約束之修正
- **2026-06-13**：修正 `report_reason` 之資料庫 check constraint，使其接受 `community_watch_*` 系列理由（`388215aad`，PR #4844）。此修正解決了群眾／自動處置所同步寫入之檢舉理由與資料庫約束不一致的問題。

### 5.4 複合判定—分層告警（三層框架）
本期最重要的處置設計革新，源於一項以直連 production 資料所得之關鍵發現（見 §6.2）：**單一分數不足以作為處置依據**。據此建立三層框架：

| 層級 | 判定規則 | 處置 |
|---|---|---|
| A（自動） | 分數≥門檻 ∧ 含聯絡方式 ∧ 含招攬詞 | 高信度色情／招攬／賣帳號／博弈，可自動處置 |
| B（ring） | 作者近重複內容（≥3 篇/30 天，輪替聯絡 ID 正規化後比對） | 機械式樣板灌水 |
| C（人工） | 高分但非 A／B | 創作、意見、短回覆 → 送人工，**永不自動處置** |

- **2026-06-16**：三層分流於 `matters-server` 實作（`995810e97`，PR #4851），純判定邏輯置於獨立可測模組（regex 加 char-3gram Jaccard），對真實 production 高分樣本 12/12 驗證通過；隨後修正 ring 正規化對輪替聯絡 ID 之處理（`e1879a05a`）。三層**全程 notify-only**，重用 §5.1 之 `report-alert` Telegram 管線（新增來源 `spam_detection`），由旗標 `MATTERS_COMMENT_SPAM_ALERT`（預設關）控制，**絕不隱藏留言**。

### 5.5 自動打掃與 ring gate
- **2026-06-15（`matters-coastguard-bot`）**：海巡啟用 Phase 2 自動打掃（`communityWatchRemoveComment`），其觸發條件除分數門檻外，新增 **ring gate**（軸一 D）：自守望相助移除歷史建立模板家族索引，候選留言命中達門檻筆數之已確認類似移除即判定為已知 ring 並打掃，**不依賴模型分數**——專門補抓逐則內容評分所漏的重複性濫用（`8b7adf2`）。隨後加入近似比對與繁簡折疊（OpenCC，`112d041`）以提升對輪替變體之穩健度。
- **判準修正**：曾短暫將文章層 ring 關鍵字（律師／翻譯／貸款等）誤加入留言掃描（`126b4bf`），經辨明「文章層 ring 與留言樣態不同」後回退（`45261d2`）——記錄於此以示判準之邊界。

## 6. 關鍵實證發現

本期透過直連 production 唯讀資料（以 VPC 內唯讀查詢 Lambda，僅允許 SELECT），對處置門檻進行了實證校準，所得發現直接形塑了 §5.4 之三層設計。

### 6.1 線上門檻與分布
- 線上自動折疊門檻實為 **0.94**（`feature_flag.spam_detection.value`），而非設計初期所假設之 0.8。
- 過去七日約 1,620 則留言被評分，分數 ≥0.9 者僅 24 則——高分濫用為**低頻、長尾**事件。

### 6.2 純分數判定之精準度不足
即使在 0.94 門檻，純分數判定之精準度僅約 **60%**。在 20 則 ≥0.94 之留言中，約 6–7 則為誤判，包括中文連載小說（創作）、日常短回覆、真實道謝、向站方之建議、以及遊戲哲學評論。進一步發現：**分數與帳號年齡均無法單獨區分真偽**（一 escort 帳號已存在 818 日、發文 883 篇）。

### 6.3 能零誤殺區分之複合訊號
經實證，唯一能在真實高分樣本上**零誤判**地區分「真正的垃圾／濫用／色情」者，為三訊號之合取：**分數 ∧ 聯絡方式（電話／Telegram／賴／外部連結）∧ 招攬詞（外送茶／約妹／博弈／賣帳號）**。任一單訊號皆會過殺（例如 @提及會誤觸「聯絡方式」、「下注」會誤觸「招攬詞」），唯合取方精準。此即 §5.4 之 A 層設計依據。此外亦確認海巡 bot 帳號為一般使用者權限、無法讀取 OSS 分數，故分數漏斗（score-funnel）必須置於伺服器端而非 bot。

## 7. 訓練資料留存與基礎設施（軸二）

為使處置體系能**持續學習、對抗樣板漂移**，本期建立了三層訓練資料管線（`spam-detection-scaffold`，roadmap 見 `3ecccf3`）。

- **L1（被動增量抽取，`b09a8b8`）**：以 SQL 自資料庫抽取訓練樣本——守望相助未被推翻之處置為正樣本（spam），被推翻／還原者為 hard-negative（ham）。此設計使**群眾與站方的每一次判斷皆轉化為標註**。
- **L2（主動快照，防硬刪，`matters-server` `9d44f315c` PR #4846 + scaffold `4603589`）**：matters-server 於處置當下將去識別之樣本送 SQS（emit 時即以 HMAC 去識別），由 Lambda worker 寫入 S3，與營運資料庫之刪除工具解耦。
- **L3（組裝與標籤品質，`460e3db`）**：去重（comment_hash）、ham 覆蓋 spam、依來源強度排序、並以 label_weight 壓低誤殺風險。
- **L2 基礎設施（2026-06-14，`c027a3f`）**：S3 桶（365 日 lifecycle）、SQS 加死信佇列、worker Lambda、SSM 參數均已建置並通過 smoke-test。
- **保留期（2026-06-14，`94b5bba`）**：訓練資料保留期定為 **1 年（365 日）**，法務方向已過審。
- **VPC 內執行入口（2026-06-15，`19397a4`）**：因分析用讀取複本（read-replica）位於 VPC 內、外部 runner 無法連線，建立 CodeBuild-in-VPC（`spam-vpc-runner`）以執行 replica-bound 工項（ring SQL、模型驗收、L1 匯出）。

去識別（HMAC）、有限保留（1 年）、與營運刪除工具之解耦，共同構成訓練資料留存之**隱私治理**基礎。

## 8. 模型更新：標籤偏差之矯正與重訓

雖然本次重訓主要驅動自文章層之驗收，但其所揭示之偏差性質、所採用之矯正方法、以及所沉澱之資料治理範式，對留言層具同等的方法論意義，故納入本報告。

### 8.1 偏差之發現與定性
對「訓練截點之後、模型未曾見過」之乾淨樣本進行驗收時發現：舊模型以 **0.98–1.0 之高信度**將華語政治評論、學術文章、個人創作判為垃圾。根因為訓練資料採用之**代理標籤**——「作者被限制（`user_restriction`）→ 其名下全部文章標為 spam」。此一「連坐」式標註，使大量合法表達被掃入正樣本，模型遂習得「華語長篇／異議書寫 ≈ 垃圾」之危險捷徑；對外語 SEO 垃圾反而判別精準。偏差因此具有**系統性方向**：對內傷己、對外精準。抽樣審計顯示，「乾淨外觀」之 spam 標註切片約有 40–55% 為誤標之合法內容。

### 8.2 矯正方法：人在環內的規模化標註
為矯正數萬筆連坐誤標，且避免將用戶內容送往外部 API，本期採用**平台同一信任邊界內的 AI 代理平行標註**：主控訂定判準與抽查把關，多個 Claude subagent 分批逐篇判定（`spam-detection-scaffold` `dca…`/`dc824b0` 之 LLM-judge 標註引擎與驗證 harness）。四輪累計產出 **4,980 筆矯正標籤（2,649 筆 hard-negative ham + 2,331 筆 spam）**。

### 8.3 重訓哲學與成效
採「**只修標籤、不純做類別平衡**」之策略（先前純平衡之嘗試曾將模型整體洗鈍）。重訓後 v1 模型（XLM-R，驗證 ROC-AUC 0.969）之驗收結果：在 625 篇人工確認之乾淨 held-out 上，**真實誤殺率由 24.0% 降至 1.6%（@0.5），spam recall 維持 92%**；12 篇人工試金石中，政治評論、學術、日記之誤殺全數修復，真垃圾辨識完全保留。

### 8.4 部署
- **2026-06-16（`spam-detection-serverless`）**：重建 conformal 校準（`3117756`）、建立帶防呆與失敗回滾之 production 促進 workflow（`bb18dc7`），v1 模型經 staging 驗證後**上線 production**（PR #6，`7996fa0`）。線上即時生效：舊模型誤殺之政治／學術內容自高分降至 0.03–0.33（不再被埋沒），真垃圾仍 ≥0.95（照常排除）。

此處所確立之原則——**面對系統性偏差應修因（重訓）而非修果（加重後處理）**，以及**唯有逐篇確認之乾淨樣本所得之數字方可信**——同樣指導留言層之門檻校準與三層設計。

## 9. 方法論之教訓（踩過的雷）

本體系建構過程中，數個關鍵轉折皆因「先質疑數字、再下結論」而免於誤判。茲彙整如下，以為後續之鑑：

1. **量測管線假象（payload 格式）**：將 JSON `{"text":...}` 送予僅接受 raw body 之文章 endpoint，模型會將 JSON 字串本身評分，恆回約 1.0，製造「100% 誤殺」之假象（scaffold `b8263e8` 修正）。教訓：異常到不合理的數字，應先懷疑量測管線。
2. **代理標籤污染驗收集**：以「老帳號、未受限」作為「乾淨合法內容」之代理，所得誤殺率達 36%；逐篇複核後發現該代理約 37% 實為垃圾（老帳號亦接外送茶、做貸款 SEO），去污染後真值僅 1.6%（scaffold `b973cb2`）。教訓：在被污染之標籤上計算之「誤殺率」嚴重失真。
3. **偽裝垃圾**：貸款／券商／討債帳號所發之「防詐／理財科普」、結尾導流至自家產品者，為偽裝成知識之 SEO 垃圾，曾被誤標為合法。判準須兼看「發文帳號性質＋結尾導流意圖」，而非僅看文體。
4. **單一訊號必然過殺**：以 @提及作為聯絡訊號會誤觸正當回覆，以「下注」作為招攬詞會誤觸自述虧損之評論——唯合取（AND）方精準（§6.3）。
5. **ring 正規化須去除輪替之聯絡 ID**：垃圾樣板輪替含數字之聯絡 ID（如 `sk3826`）；僅清純數字會殘留字母而比對失敗，須整塊清除含數字之英數 token，同時保留純字母單字（`matters-server` `e1879a05a`）。
6. **基礎設施與權限之現實摩擦**：CodeBuild ENI 無公有 IP，須排除無 NAT 之公有子網（scaffold `e87e094`）；SageMaker 角色對訓練桶無讀取權限須搬桶；read-replica 之昂貴查詢會被 recovery 取消須重試（`b5453c5`）；卡死之 CloudFormation stack 須刪除重建。
7. **機密外洩**：HF token 於 `spam-detection-serverless` git 歷史（`dc4c209`）洩漏，已委外輪替處理中（見 §11）。

共同主題：**自動化內容治理最大的風險，不在模型不夠強，而在評估時被自身的代理指標所欺，誤以為「擋掉很多」即為「做對了」。**

## 10. 治理與用戶權益

綜觀本年度之建構，可萃取出一套貫穿始終的治理姿態，建議明文制度化：

1. **誤殺優先級高於漏抓**：於異議寫作平台，壓制正當表達之代價遠大於放過垃圾；所有門檻與處置一律向「寧可漏抓、不可誤殺」傾斜。
2. **先告警、後處置（notify-first / shadow）**：留言三層分流上線即為 notify-only，先將「系統將處置什麼」攤予人覆核，確認 A 層精準後方考慮升級為自動處置。
3. **人在環內**：低信度與無法零誤殺判定者（C 層、灰區）一律送人工。
4. **可逆與可救濟**：群眾與自動處置皆可還原、可申訴；處置上限為折疊（不刪除），非刪除。
5. **群眾參與與站方制衡並存**：守望相助下放部分處置權予社群，同時以站方覆核與全程審計防止濫權與誤判。
6. **隱私治理**：訓練樣本於 emit 當下即去識別（HMAC），保留期限 1 年、法務過審，並與營運刪除工具解耦。
7. **修因不修果**：面對系統性偏差，選擇重訓矯正標籤，而非以更激進之後處理掩蓋。
8. **處置帶理由**：conformal 與群眾處置均附可讀理由，呼應 DSA 第 17 條「處置理由聲明」之精神。

## 11. 結論與後續工作

2026 年上半年，Matters 之留言濫用處置體系由「文章層反垃圾基礎之延伸」，發展為一個**多層次、群眾參與、可問責、且具自我改善能力**的治理體系：守望相助將社群納入處置並以站方制衡；專用留言模型與海巡 bot 提供自動覆蓋；複合判定—三層告警框架以「先觀察、零誤殺」為原則處理高分留言；三層訓練資料管線與一次關鍵的標籤偏差矯正，則確保體系能持續對齊平台價值並抵抗漂移。

**目前線上狀態**：留言模型評分、海巡 ring 自動打掃、留言三層分流告警（旗標待開）、L2 訓練資料擷取，均已於 production 運行；留言原生自動折疊維持關閉。

**後續工作**：
1. 由營運開啟 `MATTERS_COMMENT_SPAM_ALERT`，觀察 A 層於 production 之精準度，再決定是否升級為自動處置。
2. 完成 HF token 輪替並驗證舊 token 失效（§9.7）。
3. 軸一 C：伺服器端讀取文章 conformal 之 decision，接入人工覆核佇列。
4. 持續以 L1／L2／L3 累積真實處置資料，定期重建模型校準以對抗樣板漂移——反垃圾為長期對抗，非一次性專案。
